## Dify 做了什么 vs 本地代码做了什么

```
Dify 做的事（"大脑"）               本地代码做的事（"手脚"）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  LLM 调用（分镜生成、拆分）          数据库 CRUD（项目/段落/分镜）
  工作流编排（流程控制）              文件存储管理（目录结构、版本文件）
  条件分支 / 循环                    版本追踪（shot_versions 表）
  调用外部引擎                       批量待处理项管理
  人工审核节点                       视频合成（FFmpeg）
  API 暴露和 Web 界面                 字幕生成（SRT/ASS）
                                    转场 / 片头 / 背景音乐混合
```

**直白地说：本地代码完成了两件事——数据管家和视频工厂。**

## 本地代码的具体价值

### 第一件事：数据管家（项目管理服务）

Dify **没有能力**做这些事：

```
1. 结构化数据持久化
   · 一部小说 = 1个项目 = N个段落 = M个分镜 = K个版本
   · 这种多层级关联数据，Dify 的变量系统存不了
   · Dify 工作流执行完就结束了，下次运行拿不到上次的数据

2. 分镜版本管理
   · 用户重新生成图片 3 次，需要保留 v1、v2、v3
   · 用户可以回退到任意版本
   · Dify 做不了这件事

3. 跨工作流的数据共享
   · Workflow-A 创建了项目和分镜
   · Workflow-B 需要读取这些分镜来渲染
   · Workflow-C 需要读取分镜来重生成
   · Workflow-D 需要读取已确认的分镜来合成
   · 没有统一的数据层，四个工作流之间无法共享数据

4. 状态管理
   · 哪些分镜已渲染？哪些待重生成？哪些已确认？
   · 段落进度是多少？项目整体进度是多少？
   · 这些状态查询和更新，需要一个持久化的数据层

5. 批量处理追踪
   · 用户改了 5 个分镜的台词，标记为"待处理"
   · 用户点击"批量处理"，系统一次性重生成
   · 这需要数据库来追踪 pending 状态
```

**如果去掉项目管理服务，Dify 工作流之间就没有共享的数据存储，每次都是孤立执行。**

### 第二件事：视频工厂（合成服务）

Dify **完全做不了**这些事：

```
1. 图片 → 视频
   · Ken Burns 效果（缓慢缩放平移让静态图有动感）
   · 按目标分辨率裁剪缩放
   · 控制每帧持续时长

2. 多片段拼接 + 转场
   · xfade 转场效果（淡入淡出、溶解、滑动等）
   · 精确的偏移量计算
   · 音频同步拼接

3. 字幕叠加
   · 生成 SRT/ASS 字幕文件
   · 按配音时间轴对齐
   · 烧录到视频中

4. 片头 / 背景音乐
   · 生成片头视频
   · 混合背景音乐
   · 音量调节

5. FFmpeg 命令编排
   · 复杂的 filter_complex 链
   · 多输入多输出映射
   · 错误回退（如 xfade 失败回退到 concat）
```

**Dify 是工作流引擎，不是视频处理引擎。FFmpeg 的复杂操作必须由本地代码完成。**

## 如果去掉本地代码会怎样

```
假设只用 Dify，不用本地服务：

问题 1：数据存哪里？
  · Dify 变量是临时的，工作流结束就没了
  · 下次打开 Dify，不知道上次处理到哪了
  · 没有项目列表、没有进度查询

问题 2：工作流之间怎么传数据？
  · Workflow-A 生成了分镜，Workflow-B 怎么拿到？
  · 只能靠用户手动复制粘贴数据？
  · 或者每次都从头执行？

问题 3：视频怎么合成？
  · Dify 没有 FFmpeg 节点
  · Dify 没有图片裁剪、音频混合、字幕烧录能力
  · 只能靠 HTTP 调用外部服务 —— 这就是我们的合成服务

问题 4：版本管理怎么做？
  · 用户重生成了 3 次图片
  · 想回退到第 2 版
  · Dify 里没有版本概念

问题 5：批量处理怎么做？
  · 用户改了 5 个分镜
  · 想一次性处理
  · Dify 工作流是一次性执行的，没有"待处理队列"的概念
```

## 修正后的架构理解

```
正确的理解不是"Dify 做了大部分事"，而是：

  Dify = 调度员（决定做什么、按什么顺序做、调用谁）
  本地服务 = 专业工人（实际干活：存数据、管文件、合成视频）

  没有调度员 → 工人不知道干什么
  没有工人   → 调度员只能指挥，没人干活

两者缺一不可。
```

更准确的分工图：

```
┌──────────────────────────────────────────────────────┐
│  Dify 负责                                             │
│                                                      │
│  · "这段文字怎么切分成段落？" → 问 LLM               │
│  · "这个分镜的画面描述是什么？" → 问 LLM              │
│  · "用户选了自动拆分还是手动？" → 流程分支            │
│  · "该调用生图还是 TTS？" → 流程编排                  │
│  · "图片不满意要重生成吗？" → 人工审核                │
│  · "先渲染再合成还是直接合成？" → 流程控制            │
│                                                      │
│  本质：决策和调度                                      │
└────────────────────┬─────────────────────────────────┘
                     │ HTTP 调用
                     ▼
┌──────────────────────────────────────────────────────┐
│  本地代码负责                                          │
│                                                      │
│  · "项目、段落、分镜的数据存到数据库"                  │
│  · "图片文件存在哪个目录？v1 还是 v2？"               │
│  · "这个分镜有 3 个版本，当前用第 2 版"               │
│  · "这 5 个分镜标记为待重生成"                         │
│  · "把 6 张图 + 6 段音频 + 字幕合成一个 MP4"          │
│  · "加片头、加转场、加背景音乐"                       │
│                                                      │
│  本质：存储和执行                                      │
└──────────────────────────────────────────────────────┘
```

## 总结

| 问题 | 答案 |
|:---|:---|
| 本地代码完成了什么？ | **数据持久化 + 版本管理 + 视频合成** |
| 为什么不能全放 Dify 里？ | Dify 没有持久化存储、没有 FFmpeg、工作流之间无法共享状态 |
| 本地代码量大吗？ | 不大，但每一块都是 Dify 做不了的硬需求 |
| 两者的关系？ | Dify 是大脑，本地代码是手脚，分工明确，缺一不可 |
